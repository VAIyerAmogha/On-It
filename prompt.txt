1. Bug: nothing renders after uploading a scanned/image PDF

Symptom: uploading a scanned (non-native-text) PDF shows no visible
result in the UI — no error, no content, just nothing.

Do not guess-fix this. Root-cause it first, then fix. Likely places
the failure could actually be, in order of likelihood — check each rather
than assuming:


Backend: does is_native_pdf() correctly return False for this file,
and does extract_scanned_pdf() actually get called? Add a temporary
log line if needed to confirm which path fires. A scanned PDF wrongly
classified as native would call extract_native_pdf(), get empty text,
and raise IngestionError — check for that first.
Backend: is the OCR.space call actually succeeding? Check for
IsErroredOnProcessing: true in the raw response, expired/invalid
OCR_SPACE_API_KEY, a request timeout (OCR_TIMEOUT_SECONDS), or a
file-size/free-tier limit rejection from OCR.space. Log the raw
response body on failure instead of only raising a generic
IngestionError — the real reason needs to be visible somewhere
(server logs at minimum).
Backend: when ingestion raises, does process_ingestion() actually
catch it and set extraction_status: "failed" with an error message
field on the contract doc, or does the exception just die inside the
BackgroundTask silently (a known FastAPI BackgroundTask gotcha —
exceptions inside them don't surface anywhere unless you explicitly
catch and persist them)? If there's no extraction_error field being
set, add one now: {"extraction_status": "failed", "extraction_error": str(exc)}.
Frontend: does the upload-polling code handle extraction_status: "failed" at all? If the polling loop only checks for
!== "processing" and then unconditionally navigates to the contract
page assuming success, a "failed" status would either navigate to a
contract with no milestones and nothing rendered, or hang forever if
the loop only exits on "extracted" specifically. Fix the polling
logic to explicitly branch on "failed" and render the actual
extraction_error message inline, with a retry/re-upload option — per
the original Phase 7 spec, this should already have been an inline
error state, not a silent dead end.


Fix whichever of these is the actual root cause (likely more than one —
a silent backend failure AND a frontend that doesn't handle the failed
state are both real bugs even if only one is the "primary" cause). Test
with a real scanned PDF afterward and confirm you get either (a) correctly
extracted content, or (b) a visible, specific error message — never a
blank screen.
